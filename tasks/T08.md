# T08 — [Phase 1] — Block physics bodies

## Overview
This task bridges the game's block data model with the Matter.js physics simulation. Each block that is placed in the world needs a corresponding physics body so gravity, friction, and collision work correctly. This module maintains the authoritative mapping between block IDs and their bodies, and provides the sync function that copies physics-computed positions back to the block objects the renderer reads.

## Scope
Implement `src/physics/blockBodies.ts` to manage a `Map<string, Matter.Body>` keyed by block ID. Export `createBody(block)` which creates a 32×32 Matter.js rectangle body using the physics parameters from `BLOCK_PHYSICS[block.type]` and adds it to the physics world. Export `destroyBody(blockId)` which removes the body from the world and the map. Export `getBody(blockId)` for external queries. Export `syncBodyToBlock(block)` which copies the body's current `position.x/y` back to `block.x/block.y` — this is how physics-driven movement is reflected in the renderer each frame. Handle the `isWet` flag: if `block.isWet` is `true`, override `friction` with the `wet_sand` value regardless of the block's actual type.

## Output Files
- `src/physics/blockBodies.ts` — body lifecycle management and position sync

## Depends On
- T07 — needs the `world` instance to add/remove bodies via `World.add`/`World.remove`
- T01 — needs `BLOCK_PHYSICS`, `BLOCK_SIZE`, `Block`, and `BlockType`

## Implementation Details

### Imports and body map

```typescript
// src/physics/blockBodies.ts
import Matter from 'matter-js';
import { world } from './engine';
import { BLOCK_PHYSICS, BLOCK_SIZE, Block, BlockType } from '../constants/blocks';

const { Bodies, World } = Matter;

// Maps block.id → Matter.Body
const bodyMap = new Map<string, Matter.Body>();
```

### createBody

The body origin in Matter.js is the center of the rectangle, so the center is at `(block.x + BLOCK_SIZE/2, block.y + BLOCK_SIZE/2)`. The `label` is set to `block.id` so collision callbacks can identify which block was involved.

If `block.isWet` is `true`, friction is overridden with the `wet_sand` friction value. This lets any block type behave like wet sand when soaked by the bucket or caught in a tide.

```typescript
export function createBody(block: Block): Matter.Body {
  const physics = BLOCK_PHYSICS[block.type];
  const friction = block.isWet ? BLOCK_PHYSICS['wet_sand'].friction : physics.friction;

  const body = Bodies.rectangle(
    block.x + BLOCK_SIZE / 2,  // center x
    block.y + BLOCK_SIZE / 2,  // center y
    BLOCK_SIZE,
    BLOCK_SIZE,
    {
      mass: physics.mass,
      friction,
      frictionAir: 0.01,       // light air resistance to dampen oscillations
      restitution: physics.restitution,
      label: block.id,          // used by collision handler to look up the block
      frictionStatic: friction * 1.2, // static friction slightly higher than kinetic
    },
  );

  World.add(world, body);
  bodyMap.set(block.id, body);
  return body;
}
```

### destroyBody

```typescript
export function destroyBody(blockId: string): void {
  const body = bodyMap.get(blockId);
  if (!body) return;
  World.remove(world, body);
  bodyMap.delete(blockId);
}
```

### getBody

```typescript
export function getBody(blockId: string): Matter.Body | undefined {
  return bodyMap.get(blockId);
}
```

### syncBodyToBlock

Reads the body's current physics position and writes it back to the block object so the renderer always displays blocks at their physics-computed position.

```typescript
export function syncBodyToBlock(block: Block): void {
  const body = bodyMap.get(block.id);
  if (!body) return;
  // Body position is the center; convert back to top-left corner
  block.x = body.position.x - BLOCK_SIZE / 2;
  block.y = body.position.y - BLOCK_SIZE / 2;
}
```

### syncAllBodies

A convenience function to sync every tracked body in one call — used by the render loop.

```typescript
export function syncAllBodies(blocks: Map<string, Block>): void {
  for (const [id, block] of blocks) {
    syncBodyToBlock(block);
  }
}
```

### Updating wetness on an existing body

When a block transitions between dry and wet (e.g. tide comes in), its body friction must be updated in-place:

```typescript
export function updateBodyWetness(blockId: string, isWet: boolean): void {
  const body = bodyMap.get(blockId);
  if (!body) return;
  const block = /* caller must pass type */ undefined; // see note below
  // Matter.js does not have a direct setFriction; mutate the property directly
  // (supported by Matter.js internals; body.friction is a plain number)
  body.friction = isWet
    ? BLOCK_PHYSICS['wet_sand'].friction
    : BLOCK_PHYSICS[body.label as BlockType]?.friction ?? 0.4;
}
```

> Note: `updateBodyWetness` is called by the erosion/tide system (T23) and the bucket system (T26). The caller is responsible for also updating `block.isWet` on the `Block` object and persisting the change via `supabase/blocks.ts`.

## Key Constraints
- `createBody` must be called after the block is added to the renderer (`addBlock`) so that both systems have a consistent view of the block from the same frame.
- `syncBodyToBlock` must be called every frame for every body that is not static — do not call it only on collision, as bodies move continuously under gravity and momentum.
- Bodies for static decorations (e.g. shells) should be created with `isStatic: true` once placed to reduce simulation load; the dynamic→static transition is triggered when `block.health < 0` (handled in T23).
- `frictionAir: 0.01` prevents blocks from drifting indefinitely when placed on a slope; increase if blocks slide too far on wet sand slopes.
- The `label` field on the body must equal `block.id` exactly — collision callbacks (T07's `onCollision` hook) use `body.label` to look up the corresponding block.
- `bodyMap` must be kept in sync with the renderer's `blockMap`; orphaned bodies (in physics but not renderer, or vice versa) will cause visual drift.
