# T05 — [Phase 1] — Chunk manager

## Overview
The full game map can span tens of thousands of pixels horizontally, far too large to hold every block in memory at once. This task implements a chunk-based streaming system that loads block data from Supabase as the camera scrolls into new regions and unloads data for regions the player has scrolled away from, keeping memory and query volume bounded.

## Scope
Implement `src/canvas/chunkManager.ts` with a fixed chunk width of 256 pixels. On every camera update (called from the render loop), compute which chunk indices overlap the current viewport plus a one-chunk lookahead buffer on each side. Diff this set against the previously visible set: for newly visible chunks call `loadChunk(id)` which queries Supabase for blocks with `x BETWEEN chunkStart AND chunkEnd`; for chunks that have left the extended viewport call `unloadChunk(id)` which removes those blocks from the renderer. Guard against duplicate fetches with a `Set<number>` of loaded chunk IDs. Export `updateVisibleChunks(offsetX, viewportWidth, scale)` to be called by the render loop each frame, and `getLoadedChunks()` for debugging.

## Output Files
- `src/canvas/chunkManager.ts` — chunk visibility tracking, Supabase fetch, renderer integration

## Depends On
- T04 — needs `getOffset()` and `getScale()` to compute visible world range

## Implementation Details

### Constants and chunk math

```typescript
// src/canvas/chunkManager.ts
import { supabase } from '../supabase/client';
import { addBlock, removeBlock } from './renderer';
import { Block } from '../constants/blocks';

export const CHUNK_WIDTH = 256; // pixels per chunk
const BUFFER_CHUNKS = 1;        // how many chunks beyond viewport edge to pre-load

/** Convert a world x-coordinate to the chunk index that contains it. */
export function worldXToChunkId(worldX: number): number {
  return Math.floor(worldX / CHUNK_WIDTH);
}

/** Return the world x-range [start, end) for a given chunk index. */
export function chunkIdToRange(id: number): { start: number; end: number } {
  return { start: id * CHUNK_WIDTH, end: (id + 1) * CHUNK_WIDTH };
}
```

### Chunk state tracking

```typescript
interface ChunkState {
  id: number;
  blocks: Block[];
  loading: boolean;
}

const chunkMap = new Map<number, ChunkState>();
const loadedChunkIds = new Set<number>();
let prevVisibleIds = new Set<number>();
```

### Computing visible chunk IDs

```typescript
export function getVisibleChunkIds(
  offsetX: number,
  viewportWidth: number,
  scale: number,
): Set<number> {
  const worldLeft  = offsetX / scale;
  const worldRight = (offsetX + viewportWidth) / scale;

  const firstChunk = worldXToChunkId(worldLeft)  - BUFFER_CHUNKS;
  const lastChunk  = worldXToChunkId(worldRight) + BUFFER_CHUNKS;

  const ids = new Set<number>();
  for (let id = Math.max(0, firstChunk); id <= lastChunk; id++) {
    ids.add(id);
  }
  return ids;
}
```

### Load and unload

`loadChunk` is guarded by `loadedChunkIds` so rapid camera movement cannot fire duplicate queries for the same chunk. The Supabase query uses a range filter on the `x` column; blocks that straddle a chunk boundary belong to the chunk that contains their left edge.

```typescript
async function loadChunk(id: number): Promise<void> {
  if (loadedChunkIds.has(id)) return; // already loading or loaded
  loadedChunkIds.add(id);

  const state: ChunkState = { id, blocks: [], loading: true };
  chunkMap.set(id, state);

  const { start, end } = chunkIdToRange(id);
  const { data, error } = await supabase
    .from('blocks')
    .select('*')
    .gte('x', start)
    .lt('x', end);

  if (error) {
    console.error(`[ChunkManager] Failed to load chunk ${id}:`, error.message);
    loadedChunkIds.delete(id); // allow retry on next frame
    chunkMap.delete(id);
    return;
  }

  // Map DB rows (snake_case) to Block interface (camelCase)
  const blocks: Block[] = (data ?? []).map(row => ({
    id: row.id,
    type: row.type,
    x: row.x,
    y: row.y,
    plotId: row.plot_id,
    isWet: row.is_wet,
    health: row.health,
  }));

  state.blocks = blocks;
  state.loading = false;

  // Register each block with the renderer
  for (const block of blocks) {
    addBlock(block);
  }
}

function unloadChunk(id: number): void {
  const state = chunkMap.get(id);
  if (!state) return;

  for (const block of state.blocks) {
    removeBlock(block.id);
  }

  chunkMap.delete(id);
  loadedChunkIds.delete(id);
}
```

### Main update function (called each frame from render loop)

```typescript
export function updateVisibleChunks(
  offsetX: number,
  viewportWidth: number,
  scale: number,
): void {
  const currentIds = getVisibleChunkIds(offsetX, viewportWidth, scale);

  // Load newly visible chunks
  for (const id of currentIds) {
    if (!prevVisibleIds.has(id)) {
      loadChunk(id); // async, fire-and-forget
    }
  }

  // Unload chunks that left the viewport + buffer
  for (const id of prevVisibleIds) {
    if (!currentIds.has(id)) {
      unloadChunk(id);
    }
  }

  prevVisibleIds = currentIds;
}
```

### Debugging helper

```typescript
export function getLoadedChunks(): number[] {
  return Array.from(loadedChunkIds).sort((a, b) => a - b);
}
```

### Integration with render loop

In `renderer.ts`, the existing `render()` function should call the chunk manager each frame:

```typescript

```

## Key Constraints
- `loadedChunkIds` is the single source of truth for "this chunk is loaded or being loaded" — it must be added to before the async fetch begins and removed only on fetch error or explicit unload.
- Unloading must happen synchronously on the same frame the chunk leaves the buffer zone — do not defer unloads to avoid visual pop-in on re-entry.
- Blocks that are added to the renderer by `loadChunk` and then modified by physics (T08) or carving (T20) must have their updates reflected back to the `ChunkState.blocks` array, or unloading will leave stale data in the renderer. This is a known limitation addressed in a later cleanup pass.
- Do not fetch all chunks at startup — the user may never scroll to the far end of the map.
- Chunk IDs must be non-negative integers; `worldXToChunkId` should clamp to 0 for negative world x.
- The Supabase query does not filter by `plot_id` — all blocks in the x-range are loaded regardless of ownership, so players can see each other's castles.

## Acceptance Criteria
- [ ] Scrolling rightward causes Supabase queries to fire for newly entered chunks (visible in browser Network tab).
- [ ] Scrolling left again does not re-query already-loaded chunks (no duplicate requests for the same chunk ID).
- [ ] Blocks fetched from Supabase appear in the renderer within one animation frame after the fetch resolves.
- [ ] Scrolling far left and right and back does not produce duplicate block renders.
- [ ] `getLoadedChunks()` returns a sorted array of currently loaded chunk IDs.
- [ ] A Supabase fetch error logs to console and allows the chunk to be re-fetched on the next update cycle.
- [ ] TypeScript compiles with zero errors.
