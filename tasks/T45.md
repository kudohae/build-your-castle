# T45 — [Phase 4] — Quest Completions DB & Promotion Reward

## Overview
This task wires the quest completion pipeline to the AIT promotion reward system, preventing duplicate grants via a DB uniqueness constraint and gracefully handling the budget-exhausted error code (4109) so the player's quest is still marked complete even if the reward cannot be issued.

## Scope
Create `src/ait/promotion.ts` with `grantReward(userId, questId, promotionCode, amount)` which: (1) checks `quest_completions` for an existing `(user_id, quest_id)` row and returns early if found; (2) calls `AIT.grantPromotionRewardForGame({ promotionCode, amount })`; (3) on success inserts into `quest_completions`; (4) on AIT error code `4109` (budget exhausted) logs a warning, shows a user-facing message "보상 예산이 소진되었습니다. 칭호는 지급됩니다.", and still inserts the `quest_completions` row so the quest is marked done and the title badge is granted. Other AIT errors are rethrown. Export `checkAndGrantQuests(state, userId, blocks?)` which iterates all 7 `QUESTS`, checks if each is already complete in `quest_completions`, evaluates the `check` function, and calls `grantReward` for newly-met quests. Modify `src/supabase/blocks.ts` to emit `updateGameState` events via a hook after block mutations.

## Output Files
- `src/ait/promotion.ts` — new file; `grantReward`, `checkAndGrantQuests`
- `src/supabase/blocks.ts (modify)` — adds `updateGameState` event hooks after insert/delete/carve operations

## Depends On

## Implementation Details

### quest_completions Table Schema

```sql
CREATE TABLE quest_completions (
  user_id   UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  quest_id  TEXT NOT NULL,
  granted_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (user_id, quest_id)
);

-- RLS: users can only read their own completions
CREATE POLICY "own_completions_select" ON quest_completions
  FOR SELECT USING (user_id = auth.uid());

-- RLS: service role inserts only (called from promotion.ts server-side flow)
-- For client-side trust: allow insert if user_id = auth.uid()
CREATE POLICY "own_completions_insert" ON quest_completions
  FOR INSERT WITH CHECK (user_id = auth.uid());
```

The `(user_id, quest_id)` PRIMARY KEY enforces uniqueness at the DB level as a second guard after the application-level check.

### grantReward

```typescript
export async function grantReward(
  userId: string,
  questId: string,
  promotionCode: string,
  amount: number
): Promise<{ granted: boolean; budgetExhausted: boolean }> {
  // 1. Idempotency check
  const { data: existing } = await supabase
    .from('quest_completions')
    .select('quest_id')
    .eq('user_id', userId)
    .eq('quest_id', questId)
    .maybeSingle();

  if (existing) {
    return { granted: false, budgetExhausted: false }; // already granted
  }

  const ait = getAIT();
  let budgetExhausted = false;

  try {
    await ait.grantPromotionRewardForGame({ promotionCode, amount });
  } catch (err: any) {
    const code = err?.code ?? err?.errorCode ?? 0;
    if (Number(code) === AIT_ERROR_BUDGET_EXHAUSTED) {
      // Budget exhausted: log, notify user, continue to mark quest complete
      budgetExhausted = true;
      console.warn(
        `[promotion] Budget exhausted for quest "${questId}" (code 4109). ` +
        'Marking quest complete without monetary reward.'
      );
      showBudgetExhaustedNotice();
    } else {
      // Unknown AIT error — rethrow; quest not marked complete
      throw err;
    }
  }

  // 2. Insert quest completion (regardless of budget exhaustion)
  const { error: insertErr } = await supabase
    .from('quest_completions')
    .insert({ user_id: userId, quest_id: questId });

  if (insertErr) {
    // Duplicate key = race condition: already inserted by concurrent call
    if (insertErr.code === '23505') {
      return { granted: false, budgetExhausted };
    }
    throw new Error(`퀘스트 완료 저장 실패: ${insertErr.message}`);
  }

  return { granted: true, budgetExhausted };
}

function showBudgetExhaustedNotice(): void {
  (window as any).__sandCastleToast?.(
    '보상 예산이 소진되었습니다. 칭호는 지급됩니다.',
    'warning'
  );
}
```

### checkAndGrantQuests

```typescript
/** Called by updateGameState after every relevant game event. */
export async function checkAndGrantQuests(
  state: GameState,
  blocks?: BlockGrid
): Promise<void> {
  const userId = getCurrentUserId();

  // Fetch already-completed quests in one query
  const { data: completions } = await supabase
    .from('quest_completions')
    .select('quest_id')
    .eq('user_id', userId);

  const completedIds = new Set(
    (completions ?? []).map((r: any) => r.quest_id)
  );

  for (const quest of QUESTS) {
    if (completedIds.has(quest.id)) continue; // already done

    let met: boolean;
    try {
      // Arch quest requires block grid; pass it in
      met = quest.id === 'arch'
        ? quest.check(state, blocks)
        : quest.check(state);
    } catch {
      continue; // defensive: check fn threw, skip
    }

    if (!met) continue;

    try {
      const result = await grantReward(userId, quest.id, quest.promotionCode, quest.reward);
      if (result.granted) {
        // Notify T46 UI to trigger celebration
        window.dispatchEvent(
          new CustomEvent('quest:completed', {
            detail: { questId: quest.id, title: quest.title },
          })
        );
      }
    } catch (err) {
      console.error(`[promotion] grantReward failed for ${quest.id}:`, err);
    }
  }
}
```

The completed-IDs fetch is batched into a single query per `checkAndGrantQuests` call. This is called after every game event, so the batch query prevents N+1 calls to `quest_completions`.

### Handling Concurrent checkAndGrantQuests Calls

Because `updateGameState` calls `checkAndGrantQuests` asynchronously via dynamic import, multiple rapid events (e.g., placing many blocks quickly) may trigger overlapping calls. The `(user_id, quest_id)` PRIMARY KEY in `quest_completions` and the `23505` duplicate-key check in `grantReward` make this safe — only the first successful insert wins.

## Key Constraints
- Error code `4109` from AIT must NOT prevent quest completion — the DB insert always proceeds even when the monetary reward is blocked.
- The application-level duplicate check in `grantReward` and the DB PRIMARY KEY together form a two-layer idempotency guard.
- `checkAndGrantQuests` must not throw — all per-quest errors are caught and logged so one failing quest does not block others.
- The `quest:completed` CustomEvent is fired only when `result.granted === true` (first-time completion), not on subsequent calls.
- `blocks.ts` hooks must NOT await `updateGameState` in the hot path — the call is fire-and-forget from the block operation's perspective.
- The block grid passed to `updateGameState` on `block_placed` must be the post-placement grid state, not the pre-placement state.
- AIT SDK errors other than `4109` are re-thrown — they indicate configuration problems (wrong promotion code, auth failure) that need developer attention.

## Acceptance Criteria
- [ ] `grantReward` called twice for the same `(userId, questId)` inserts only one row (idempotency).
- [ ] AIT error code `4109` results in a "보상 예산이 소진되었습니다" toast but still inserts the `quest_completions` row.
- [ ] AIT errors with codes other than `4109` are rethrown and the `quest_completions` row is NOT inserted.
- [ ] `checkAndGrantQuests` fetches all completions in a single query before iterating quests (no N+1).
- [ ] After quest completion, a `quest:completed` CustomEvent is dispatched with `questId` and `title` in `detail`.
- [ ] `digBlock` in `blocks.ts` calls `updateGameState` with `type: 'block_collected'` after a successful DB write.
- [ ] `placeBlock` passes the current block grid to `updateGameState` so arch and height checks have current data.
- [ ] `getCompletedQuestIds` returns a `Set<string>` that T46 can use to show ✅ vs 🔒 without additional queries.
