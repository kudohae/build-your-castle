# T11 — [Phase 1] — Toss login & user sync

## Overview
All gameplay is gated behind Toss authentication. This task implements the login flow using the AIT (Apps in Toss) SDK, caches the authenticated user in memory, and syncs the Toss identity into the game's own Supabase `users` table so the rest of the system has a stable internal user ID to reference. The app entry point must call `login()` before any other logic runs.

## Scope
Implement `src/ait/auth.ts` with three exports: `login()` which calls the AIT `TossLogin.login()` SDK method, extracts the `toss_user_id`, calls `syncUser()`, and caches the result; `syncUser(tossUserId)` which upserts the user into Supabase and returns the typed `User` row; `getUser()` which returns the cached user or `null` if not yet authenticated. Define the `User` interface with `id`, `tossUserId`, and `nickname` fields. The module-level `currentUser` variable holds the cached session.

## Output Files
- `src/ait/auth.ts` — login flow, Supabase user sync, in-memory session cache

## Depends On
- T02 — needs the `supabase` singleton for the users upsert

## Implementation Details

### User interface

```typescript
// src/ait/auth.ts
import { supabase } from '../supabase/client';

// Import the AIT SDK — exact import path per Apps in Toss documentation
import { TossLogin } from '@apps-in-toss/web-framework';

export interface User {
  id: string;          // Supabase users.id (UUID)
  tossUserId: string;  // Toss platform user ID (opaque string)
  nickname: string | null;
}

// Module-level cache — survives across component re-renders but resets on page reload
let currentUser: User | null = null;
```

### syncUser

Upserts the Toss user ID into the Supabase `users` table. Uses `onConflict: 'toss_user_id'` to ensure idempotency — calling this multiple times with the same ID is safe. Returns the canonical internal `User` object.

```typescript
export async function syncUser(tossUserId: string): Promise<User> {
  const { data, error } = await supabase
    .from('users')
    .upsert(
      { toss_user_id: tossUserId },
      { onConflict: 'toss_user_id' },
    )
    .select('id, toss_user_id, nickname')
    .single();

  if (error) throw new Error(`[auth.syncUser] ${error.message}`);

  const user: User = {
    id: data.id as string,
    tossUserId: data.toss_user_id as string,
    nickname: data.nickname as string | null,
  };

  currentUser = user;
  return user;
}
```

### login

Calls the AIT SDK, extracts the user ID from the response, and delegates to `syncUser`. The exact shape of the `TossLogin.login()` response depends on the AIT SDK version — consult the Apps in Toss developer documentation for the precise field name. The `toss_user_id` field shown here is the documented identifier.

```typescript
export async function login(): Promise<User> {
  // AIT SDK call — blocks until the user completes Toss authentication
  const loginResult = await TossLogin.login();

  // Extract the stable Toss user identifier
  // (field name may vary by AIT SDK version; check @apps-in-toss/web-framework docs)
  const tossUserId: string = loginResult.toss_user_id ?? loginResult.userId;

  if (!tossUserId) {
    throw new Error('[auth.login] TossLogin did not return a user ID. Check AIT SDK version.');
  }

  return syncUser(tossUserId);
}
```

### getUser

```typescript
export function getUser(): User | null {
  return currentUser;
}
```

### App entry point usage

The app's `main.ts` (or equivalent) must call `login()` before starting the renderer or loading any game data:

```typescript
// src/main.ts (integration pattern)
import { login } from './ait/auth';
import { start as startRenderer } from './canvas/renderer';
import { initCamera } from './canvas/camera';
import { getOrCreatePlotForUser } from './supabase/plots';

async function main() {
  const user = await login(); // blocks until auth complete
  const plot = await getOrCreatePlotForUser(user.tossUserId);
  startRenderer();
  initCamera();
  // ... load blocks for plot
}

main().catch(console.error);
```

## Key Constraints
- `login()` must be awaited before calling any Supabase queries that depend on the user's identity — calling `getOrCreatePlotForUser` before `login()` resolves will use `null` as the userId, which the database will reject.
- `currentUser` is intentionally module-level (not in React state or localStorage) because this game has no React layer — it is a Canvas 2D app.
- `persistSession: false` on the Supabase client (set in T02) means Supabase will not maintain a JWT session; the AIT SDK owns the auth token. If any Supabase operation requires a user-scoped JWT, the AIT SDK must provide it as a custom auth header — this integration is out of scope for Phase 1.
- Do not hardcode any Toss user IDs or test credentials — all identity flows through the SDK.
- If `TossLogin.login()` rejects (e.g. user cancels), the error must propagate to the entry point so the app can show an error UI instead of silently failing.
- `nickname` may be `null` if the Toss account has no display name configured; all UI that displays a nickname must handle the null case.

## Acceptance Criteria
- [ ] `login()` returns a `User` object with a non-empty `tossUserId` after a successful AIT auth flow.
- [ ] Calling `getUser()` before `login()` returns `null`.
- [ ] Calling `getUser()` after `login()` returns the same object as `login()`'s return value.
- [ ] `syncUser('test-user-999')` upserts into Supabase and returns a `User` with the same `tossUserId`.
- [ ] Calling `syncUser('test-user-999')` twice returns the same `id` both times (idempotent upsert).
- [ ] TypeScript compiles with zero errors; `User` interface has correct field types.
- [ ] `login()` propagates SDK errors rather than swallowing them.
