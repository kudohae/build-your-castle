# T50 — [Phase 4] — Season System & Seasonal Blocks in Store

## Overview
This task implements the season detection system and wires it into the store UI so that the season tab always shows the current season's exclusive blocks. It defines the full block mapping for all five seasons and ensures that past-season blocks owned by players remain usable in their castles — seasonal exclusivity applies only to new acquisition.

## Scope
Create `src/systems/season.ts` exporting `getCurrentSeason()`, a `SEASON_BLOCKS` mapping, and `getSeasonBlocks(season)`. `getCurrentSeason()` reads the current month and returns one of five season string literals: `'summer'` (months 7–8), `'autumn'` (months 9–11), `'christmas'` (month 12), `'new_year'` (months 1–2), `'spring'` (months 3–4); month 5–6 maps to `'spring'` (extended spring). `SEASON_BLOCKS` maps each season to its two exclusive block type strings. `getSeasonBlocks(season)` returns the block type array for that season. Modify `src/ui/store.ts` so that when `StoreScreen.open()` is called, it calls `getCurrentSeason()` and filters the season tab's `PRODUCT_CARDS` to include only the current season's block pack, updating the card's `description` to name the specific blocks. Past-season blocks already in a player's inventory remain visible in `inventory_items` and can be placed in the castle — no retroactive removal logic is implemented.

## Output Files
- `src/systems/season.ts` — new file; `Season` type, `getCurrentSeason`, `SEASON_BLOCKS`, `getSeasonBlocks`
- `src/ui/store.ts (modify)` — filters season tab on `open()`, shows current season's block names in the season pack card

### getCurrentSeason

```typescript
/**
 * Returns the current game season based on the calendar month.
 *
 * Month mapping:
 *   1–2  → new_year (January, February)
 *   3–6  → spring   (March through June — extended spring)
 *   7–8  → summer   (July, August)
 *   9–11 → autumn   (September, October, November)
 *   12   → christmas
 */
export function getCurrentSeason(): Season {
  const month = new Date().getMonth() + 1; // 1–12

  if (month === 12)               return 'christmas';
  if (month === 1 || month === 2) return 'new_year';
  if (month >= 3 && month <= 6)   return 'spring';
  if (month === 7 || month === 8) return 'summer';
  if (month >= 9 && month <= 11)  return 'autumn';

  // Fallback (should be unreachable)
  return 'spring';
}

/**
 * Returns the two block type strings for the given season.
 */
export function getSeasonBlocks(season: Season): [string, string] {
  return SEASON_BLOCKS[season];
}

/**
 * Returns the Season that corresponds to a given month (1–12).
 * Useful for testing without mocking Date.
 */
export function getSeasonForMonth(month: number): Season {
  if (month === 12)               return 'christmas';
  if (month === 1 || month === 2) return 'new_year';
  if (month >= 3 && month <= 6)   return 'spring';
  if (month === 7 || month === 8) return 'summer';
  if (month >= 9 && month <= 11)  return 'autumn';
  return 'spring';
}
```

`getSeasonForMonth` is a pure function that accepts a month integer — it enables deterministic unit testing without mocking the global `Date` object.

### Store Modification — Season Tab Filtering

```typescript
// src/ui/store.ts (modifications to StoreScreen)

import {
  getCurrentSeason,
  getSeasonBlocks,
  BLOCK_DISPLAY_NAMES,
  SEASON_LABELS,
  Season,
} from '../systems/season';

// Inside StoreScreen class:

private currentSeason: Season = 'spring';

async open(initialTab?: TabId): Promise<void> {
  this.isOpen = true;
  this.activeTab = initialTab ?? '도구 업그레이드';
  this.scrollOffset = 0;
  this.currentSeason = getCurrentSeason();
  this.updateSeasonPackCard();  // ← NEW
  await this.refreshOwnedStates();
}

private updateSeasonPackCard(): void {
  const seasonBlocks = getSeasonBlocks(this.currentSeason);
  const blockNames = seasonBlocks
    .map((b) => BLOCK_DISPLAY_NAMES[b] ?? b)
    .join(', ');
  const seasonLabel = SEASON_LABELS[this.currentSeason];

  // Find and mutate the season_pack card entry in PRODUCT_CARDS
  const seasonPackCard = PRODUCT_CARDS.find(
    (c) => c.productId === PRODUCTS.season_pack
  );
  if (seasonPackCard) {
    seasonPackCard.name = `${seasonLabel} 시즌 팩`;
    seasonPackCard.description =
      `${blockNames}\n각 10개씩 제공`;
    seasonPackCard.icon = this.getSeasonIcon(this.currentSeason);
  }
}

private getSeasonIcon(season: Season): string {
  const icons: Record<Season, string> = {
    summer:    '☀️',
    autumn:    '🍂',
    christmas: '🎄',
    new_year:  '🎍',
    spring:    '🌸',
  };
  return icons[season];
}
```

### Past-Season Block Persistence Policy

Past-season blocks remain in the player's inventory and may be placed in their castle indefinitely. No removal logic is implemented — the exclusivity only prevents new acquisition of out-of-season blocks. This is enforced by the store's season tab filtering: only the current season's pack is shown for purchase.

To verify this policy in the inventory:

### Season Change Detection on App Resume

If the app is left open across a season boundary (e.g., open at 23:59 on Nov 30, resumed at 00:01 on Dec 1), the season should refresh on next store open:

```typescript
// open() already calls getCurrentSeason() fresh on each open:
async open(initialTab?: TabId): Promise<void> {
  this.currentSeason = getCurrentSeason(); // always re-reads Date
  // ...
}
```

No session-level caching of the season is introduced — the fresh `Date()` read in `getCurrentSeason()` handles this automatically.

## Key Constraints
- `getCurrentSeason()` always derives the season from `new Date()` at call time — it is never cached at module level. This ensures accuracy across midnight season transitions.
- Months 5–6 (May–June) map to `'spring'` as an extended spring period, even though the game spec says spring is 3–4월. This is intentional to leave no uncovered months in the season mapping.
- `SEASON_BLOCKS` values are two-element tuples (`[string, string]`) typed as such — do not expand a season to more than 2 blocks without updating the `getSeasonBlocks` return type.
- The store's `PRODUCT_CARDS` array is mutated in place by `updateSeasonPackCard()` — this is acceptable because the array is module-scoped and `open()` always refreshes it before drawing.
- Seasonal block rendering colors in `SEASON_BLOCK_COLORS` must be registered in the block renderer's color lookup table (T01/renderer) — this task does not modify the renderer but provides the required data.
- Past-season block removal is explicitly out of scope — the design preserves player-owned assets indefinitely regardless of the current season.
- `getSeasonForMonth` is the test-friendly pure function; `getCurrentSeason` wraps it with `new Date()`. Unit tests should use `getSeasonForMonth` directly.
