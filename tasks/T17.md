# T17 — [Phase 2] — Block interface update for pixelMask

## Overview
Now that the `PixelMask` type exists (T16), this task wires it into the core `Block` interface and the persistence layer. Every block in the game world gains a pixel mask that the carving system (upcoming) will read and modify. Blocks loaded from Supabase that have no stored mask default to a full mask, ensuring backward compatibility with any blocks saved before this task.

## Scope
Modify `src/constants/blocks.ts` to add `pixelMask: PixelMask` to the `Block` interface. Modify `src/supabase/blocks.ts` to: serialize the mask in `saveBlock`'s upsert payload; deserialize it in `loadBlocks`'s row mapping (defaulting to `createFullMask()` for rows with no stored mask); add a new targeted function `updateBlockMask(id, mask)` that updates only the `pixel_mask` column and `updated_at`. This targeted function will be called frequently by the carving tool (T20) instead of the full `saveBlock` to minimize write bandwidth.

## Output Files
- `src/constants/blocks.ts (modify)` — adds `pixelMask: PixelMask` to the `Block` interface
- `src/supabase/blocks.ts (modify)` — adds mask serialization to `saveBlock`, deserialization to `loadBlocks`, and new `updateBlockMask` function

## Depends On
- T16 — needs `PixelMask`, `serializeMask`, `deserializeMask`, `createFullMask`
- T10 — the file being modified; `saveBlock` and `loadBlocks` are extended

### Modifying src/supabase/blocks.ts — loadBlocks

```typescript
// Add imports at top of supabase/blocks.ts:
import {
  PixelMask,
  createFullMask,
  deserializeMask,
} from '../carving/pixelMask';

// Updated rowToBlock helper:
function rowToBlock(row: Record<string, unknown>): Block {
  return {
    id: row.id as string,
    type: row.type as BlockType,
    x: row.x as number,
    y: row.y as number,
    plotId: row.plot_id as string,
    isWet: row.is_wet as boolean,
    health: row.health as number,
    pixelMask: row.pixel_mask
      ? deserializeMask(row.pixel_mask as string)
      : createFullMask(),  // backward-compatible default for pre-T17 rows
  };
}

// Updated loadBlocks — include pixel_mask in the select:
export async function loadBlocks(plotId: string): Promise<Block[]> {
  const { data, error } = await supabase
    .from('blocks')
    .select('id, type, x, y, plot_id, is_wet, health, pixel_mask')  // ← add pixel_mask
    .eq('plot_id', plotId)
    .order('created_at', { ascending: true });

  if (error) throw new Error(`[blocks.loadBlocks] ${error.message}`);
  return (data ?? []).map(rowToBlock);
}
```

### Modifying src/supabase/blocks.ts — saveBlock

```typescript
import { serializeMask } from '../carving/pixelMask';

export async function saveBlock(block: Block): Promise<void> {
  const { error } = await supabase
    .from('blocks')
    .upsert(
      {
        id: block.id,
        plot_id: block.plotId,
        type: block.type,
        x: Math.round(block.x),
        y: Math.round(block.y),
        is_wet: block.isWet,
        health: block.health,
        pixel_mask: serializeMask(block.pixelMask),  // ← NEW
        updated_at: new Date().toISOString(),
      },
      { onConflict: 'id' },
    );

  if (error) throw new Error(`[blocks.saveBlock] ${error.message}`);
}
```

### New function: updateBlockMask

Called by the carving tool after each carve stroke. Writing only `pixel_mask` and `updated_at` avoids re-sending all block fields on every brush stroke, which would be expensive at carving frequency (potentially 10–30 calls/second).

```typescript
export async function updateBlockMask(id: string, mask: PixelMask): Promise<void> {
  const { error } = await supabase
    .from('blocks')
    .update({
      pixel_mask: serializeMask(mask),
      updated_at: new Date().toISOString(),
    })
    .eq('id', id);

  if (error) throw new Error(`[blocks.updateBlockMask] ${error.message}`);
}
```

### Database migration requirement

The `pixel_mask` column must be added to the `blocks` table before this task's code is deployed. The migration SQL (to be applied via Supabase dashboard or CLI):

```sql
ALTER TABLE blocks
  ADD COLUMN IF NOT EXISTS pixel_mask text;
-- NULL = full mask (default for pre-migration rows, handled in rowToBlock)
```

No `NOT NULL` constraint is added — null rows are valid and handled in code by `createFullMask()`.

## Key Constraints
- The `pixel_mask` column is nullable — rows inserted before this migration have `NULL`, and `rowToBlock` must handle this gracefully by calling `createFullMask()`.
- `serializeMask` produces ~5460-character base64 strings. For blocks that are never carved (`isMaskFull(mask) === true`), consider storing `NULL` instead to save storage: `pixel_mask: isMaskFull(block.pixelMask) ? null : serializeMask(block.pixelMask)`. This optimization reduces storage for the majority of blocks.
- `updateBlockMask` is a write-only function for carving — it does not read the current mask from the database. The carving system (T20) maintains the in-memory mask and calls this to persist it.
- After adding `pixelMask` to `Block`, run `tsc --noEmit` to find all creation sites that need updating.
- Do not add `pixelMask` to the `ChunkManager`'s block row mapping in `chunkManager.ts` (T05) — that file also calls `rowToBlock`-equivalent logic. Ensure that mapping is also updated to include `pixel_mask` in the select and the default logic.
- The `Float32Array` in `pixelMask` is 4096 bytes in memory per block. With 1000 blocks loaded in view, this is ~4MB — acceptable for mobile, but chunk unloading (T05) is important to keep this bounded.
