# T09 — [Phase 1] — Supabase plots CRUD

## Overview
Plots are the fundamental ownership primitive of the game — each player owns one plot in the CASTLE_ZONE, and all their blocks are scoped to it. This task implements the full CRUD layer for plots, including idempotent "get or create" logic so that new players automatically receive a plot on first login without any manual provisioning step.

## Scope
Implement `src/supabase/plots.ts` with four async functions: `loadPlot(plotId)` fetches a single plot by primary key; `createPlot(ownerId, positionX)` inserts a new plot and returns the created row; `getOrCreatePlotForUser(userId)` queries by `owner_id` and calls `createPlot` with the next available `position_x` if none exists; `updateLastVisited(plotId)` sets `last_visited_at = now()`. All functions return typed results using a local `Plot` interface mirroring the database schema. All use the shared `supabase` client from T02.

## Output Files
- `src/supabase/plots.ts` — `Plot` interface and all four CRUD functions

## Depends On
- T02 — needs the `supabase` singleton client

## Implementation Details

### Plot type and schema

The `Plot` interface mirrors the database table structure. The Supabase migration (pre-existing) created this table:

```sql
-- (already applied — shown here for reference only)
CREATE TABLE plots (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  owner_id      text NOT NULL UNIQUE,  -- toss_user_id from AIT SDK
  position_x    integer NOT NULL,
  width         integer NOT NULL DEFAULT 800,
  created_at    timestamptz NOT NULL DEFAULT now(),
  last_visited_at timestamptz
);
```

```typescript
// src/supabase/plots.ts
import { supabase } from './client';

export interface Plot {
  id: string;
  ownerId: string;
  positionX: number;
  width: number;
  createdAt: string;
  lastVisitedAt: string | null;
}

/** Map a raw Supabase row (snake_case) to the Plot interface (camelCase). */
function rowToPlot(row: Record<string, unknown>): Plot {
  return {
    id: row.id as string,
    ownerId: row.owner_id as string,
    positionX: row.position_x as number,
    width: row.width as number,
    createdAt: row.created_at as string,
    lastVisitedAt: row.last_visited_at as string | null,
  };
}
```

### loadPlot

```typescript
export async function loadPlot(plotId: string): Promise<Plot | null> {
  const { data, error } = await supabase
    .from('plots')
    .select('*')
    .eq('id', plotId)
    .single();

  if (error) {
    if (error.code === 'PGRST116') return null; // row not found
    throw new Error(`[plots.loadPlot] ${error.message}`);
  }

  return rowToPlot(data);
}
```

### createPlot

```typescript
export async function createPlot(ownerId: string, positionX: number): Promise<Plot> {
  const { data, error } = await supabase
    .from('plots')
    .insert({ owner_id: ownerId, position_x: positionX })
    .select()
    .single();

  if (error) throw new Error(`[plots.createPlot] ${error.message}`);
  return rowToPlot(data);
}
```

### getOrCreatePlotForUser

Queries for an existing plot by `owner_id`. If found, returns it immediately. If not found, calculates the next available `position_x` by finding the rightmost existing plot and adding its width, then calls `createPlot`. A simple race condition (two new users onboarding simultaneously) is acceptable at this stage — the `owner_id` UNIQUE constraint ensures only one plot is created per user, and the second insert will fail gracefully. In production, a database function (RPC) should handle this atomically.

```typescript
const PLOT_WIDTH = 800;
const CASTLE_ZONE_START = 1200;

export async function getOrCreatePlotForUser(userId: string): Promise<Plot> {
  // 1. Try to find existing plot
  const { data: existing, error: fetchError } = await supabase
    .from('plots')
    .select('*')
    .eq('owner_id', userId)
    .maybeSingle();

  if (fetchError) throw new Error(`[plots.getOrCreate] ${fetchError.message}`);
  if (existing) return rowToPlot(existing);

  // 2. Find next available position_x
  const { data: rightmost, error: maxError } = await supabase
    .from('plots')
    .select('position_x, width')
    .order('position_x', { ascending: false })
    .limit(1)
    .maybeSingle();

  if (maxError) throw new Error(`[plots.getOrCreate] ${maxError.message}`);

  const nextX = rightmost
    ? (rightmost.position_x as number) + (rightmost.width as number)
    : CASTLE_ZONE_START;

  // 3. Create plot (may fail with unique violation if racing; caller should retry)
  return createPlot(userId, nextX);
}
```

### updateLastVisited

```typescript
export async function updateLastVisited(plotId: string): Promise<void> {
  const { error } = await supabase
    .from('plots')
    .update({ last_visited_at: new Date().toISOString() })
    .eq('id', plotId);

  if (error) throw new Error(`[plots.updateLastVisited] ${error.message}`);
}
```

## Key Constraints
- `owner_id` is the Toss user ID string (not the Supabase auth UUID) — T11 stores the Toss ID in the `users` table, but plots reference it directly for simplicity.
- `getOrCreatePlotForUser` is not atomic — in a multi-user environment with many simultaneous new signups, the `UNIQUE` constraint on `owner_id` will catch races. Handle the `23505` Postgres error code by retrying with a fresh read.
- `PLOT_WIDTH = 800` must match the value in the `plots` schema default so that `position_x` arithmetic is consistent.
- `CASTLE_ZONE_START = 1200` must match `MAP_ZONES` in `src/constants/maps.ts`; if that constant changes, update this file too.
- All errors must be thrown (not swallowed) so that the calling code can display user-facing error messages.
- `rowToPlot` must not be called on `null` — always check for null data before mapping.
