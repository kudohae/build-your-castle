# T41 — [Phase 4] — In-App Purchase Setup & Product Constants

## Overview
This task bootstraps the in-app purchase system by defining all purchasable product IDs as typed constants, implementing the core `purchase()` flow that calls the AIT InAppPurchase SDK, and persisting purchase receipts to Supabase for server-side verification and reward application.

## Scope
Create `src/ait/payment.ts` as the single entry point for all in-app purchase operations. Define a `PRODUCTS` const object mapping 11 human-readable keys to their AIT product ID strings (e.g. `PRODUCTS.shovel_lv2 = 'sand_castle.shovel_lv2'`). Implement `purchase(productId: ProductId): Promise<PurchaseResult>` which calls `AIT.InAppPurchase.purchase({ productId })`, and on success calls `applyPurchase(productId, receipt)` (implemented in T42) and inserts a row into the `purchases` Supabase table with fields `user_id, product_id, receipt, applied, created_at`. On SDK failure it rethrows a typed `PurchaseError`. Export the `PurchaseResult` interface and the `ProductId` union type derived from `PRODUCTS` values.

## Output Files
- `src/ait/payment.ts` — new file; product constants, `purchase()` function, `PurchaseResult` interface, `purchases` table persistence

## Depends On
- T11 — establishes the AIT SDK integration setup (auth, SDK initialization) and the Supabase client with authenticated user session

### Purchases Supabase Table Schema

```sql
CREATE TABLE purchases (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  product_id  TEXT NOT NULL,
  receipt     TEXT NOT NULL,
  applied     BOOLEAN NOT NULL DEFAULT FALSE,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- RLS: users can only see their own purchases
CREATE POLICY "own_purchases_only" ON purchases
  FOR SELECT USING (user_id = auth.uid());

-- RLS: insert only for authenticated user's own row
CREATE POLICY "insert_own_purchase" ON purchases
  FOR INSERT WITH CHECK (user_id = auth.uid());
```

### purchase() — Core Flow

```typescript
export async function purchase(productId: ProductId): Promise<PurchaseResult> {
  const ait = getAIT();
  const userId = getCurrentUserId(); // throws if not logged in

  let receipt: string;

  try {
    const result = await ait.InAppPurchase.purchase({ productId });
    receipt = result.receipt;
  } catch (err: any) {
    // Map AIT SDK error codes to PurchaseError
    const code = mapAITErrorCode(err?.code ?? err?.message ?? '');
    throw new PurchaseError(
      err?.message ?? '결제에 실패했습니다.',
      code
    );
  }

  // Persist receipt BEFORE applying (so it's recoverable if apply fails)
  const { data: purchaseRow, error: dbErr } = await supabase
    .from('purchases')
    .insert({
      user_id:    userId,
      product_id: productId,
      receipt,
      applied:    false,
    })
    .select('id, created_at')
    .single();

  if (dbErr || !purchaseRow) {
    // Receipt obtained but not persisted — log for manual recovery
    console.error('[payment] Receipt persist failed', { productId, receipt, dbErr });
    throw new PurchaseError('영수증 저장에 실패했습니다.', 'UNKNOWN');
  }

  // Apply product effects (implemented in T42)
  await applyPurchase(productId, receipt);

  // Mark as applied
  await supabase
    .from('purchases')
    .update({ applied: true })
    .eq('id', purchaseRow.id);

  return {
    success: true,
    productId,
    receipt,
    appliedAt: new Date().toISOString(),
  };
}

function mapAITErrorCode(
  raw: string
): PurchaseError['code'] {
  if (raw.includes('cancel') || raw.includes('CANCEL')) return 'USER_CANCELLED';
  if (raw.includes('ALREADY') || raw.includes('owned'))  return 'ALREADY_OWNED';
  if (raw.includes('network') || raw.includes('NETWORK')) return 'NETWORK';
  if (raw.includes('fail') || raw.includes('FAIL'))      return 'PAYMENT_FAILED';
  return 'UNKNOWN';
}
```

The receipt is inserted into `purchases` with `applied = false` BEFORE calling `applyPurchase`. This ensures that if the application step crashes, the receipt is not lost — a retry mechanism can recover it.

### Purchase Recovery on App Restart

On app startup, unapplied receipts from previous sessions are re-applied:

```typescript
export async function recoverPendingPurchases(): Promise<void> {
  const userId = getCurrentUserId();
  const { data } = await supabase
    .from('purchases')
    .select('id, product_id, receipt')
    .eq('user_id', userId)
    .eq('applied', false);

  for (const row of data ?? []) {
    try {
      await applyPurchase(row.product_id as ProductId, row.receipt);
      await supabase.from('purchases').update({ applied: true }).eq('id', row.id);
    } catch (err) {
      console.warn('[payment] Recovery failed for', row.product_id, err);
    }
  }
}
```

## Key Constraints
- The AIT SDK (`window.AIT`) is provided by the Toss host environment at runtime; it is never bundled. Always access it through `getAIT()` to get a clear error if it is unavailable.
- `USER_CANCELLED` errors must NOT persist a purchases row — the user chose not to pay. The try/catch wraps the SDK call before any DB insertion.
- Receipt must be inserted with `applied = false` before `applyPurchase` is called to enable crash recovery.
- `PRODUCTS` is `as const` so `ProductId` is a string literal union — TypeScript will catch invalid product IDs at compile time.
- `recoverPendingPurchases()` must be called once during app initialization (before the main game loop starts) to handle interrupted sessions.
- Do not expose raw AIT receipt strings in logs or error messages — they are sensitive payment tokens.
- Non-consumable products (`shovel_lv2/3/5`, `plot_expand`, `inventory_expand`) should show "보유 중" in the store UI (T43) after first purchase; the `ALREADY_OWNED` error code supports this check.

## Acceptance Criteria
- [ ] All 11 product IDs are defined in `PRODUCTS` with the correct `sand_castle.*` prefix.
- [ ] `ProductId` TypeScript type is a union of all 11 product ID string literals — passing an invalid string is a compile error.
- [ ] A successful `purchase()` call inserts a row in `purchases` with `applied = true` after completion.
- [ ] A user-cancelled purchase (AIT returns cancel error) does not insert any row in `purchases`.
- [ ] If `applyPurchase` throws after receipt insertion, the row remains in `purchases` with `applied = false`.
- [ ] `recoverPendingPurchases()` re-applies any `applied = false` rows found on app start.
- [ ] `PurchaseError` is thrown with the correct `code` for cancellation, network failure, and unknown errors.
- [ ] Calling `purchase()` when `window.AIT` is undefined throws an error before any SDK or DB call.
