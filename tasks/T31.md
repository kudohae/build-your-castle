# T31 — [Phase 3] — Art score DB & API

## Overview
The art score system allows players to rate each other's sand castles on a 1–5 star scale, creating a community-driven quality signal for the leaderboard. This task establishes the database schema (art_scores table and plot_avg_scores view), implements the Supabase client functions for submitting and querying scores, and enforces a daily voting cap of 10 votes per user to prevent abuse. The SQL schema is embedded as documentation comments to guide manual migration.

## Scope
Create `src/supabase/scores.ts` with three exported functions. `submitScore(plotId: string, voterId: string, score: 1|2|3|4|5): Promise<SubmitScoreResult>` performs an upsert into the `art_scores` table (unique constraint on `plot_id + voter_id` ensures one vote per pair, with updates replacing previous votes). `getPlotScore(plotId: string): Promise<PlotScore>` queries the `plot_avg_scores` view returning `{ avg: number, count: number }`. `getTodayVoteCount(userId: string): Promise<number>` counts rows in `art_scores` where `voter_id = userId` and `created_at >= today's midnight UTC`. Also export the SQL DDL as a string constant `MIGRATION_SQL` (for documentation; not executed programmatically) to create the `art_scores` table with columns `id uuid`, `plot_id uuid FK→plots`, `voter_id uuid FK→users`, `score smallint CHECK(1-5)`, `created_at timestamptz DEFAULT now()`, `UNIQUE(plot_id, voter_id)`, and the `plot_avg_scores` view computing `AVG(score)` and `COUNT(*)` grouped by `plot_id`.

## Output Files
- `src/supabase/scores.ts` — new file; exports `submitScore`, `getPlotScore`, `getTodayVoteCount`, `SubmitScoreResult`, `PlotScore`, `MIGRATION_SQL`

## Depends On
- T02 — provides the Supabase client singleton and the TypeScript types for Supabase responses; this module imports `supabase` from `./client`

### MIGRATION_SQL documentation constant

```typescript
export const MIGRATION_SQL = `
-- Run in Supabase SQL Editor to create the art score tables.

-- Enable UUID extension if not already active
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

-- Art scores table
CREATE TABLE IF NOT EXISTS art_scores (
  id         UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  plot_id    UUID NOT NULL REFERENCES plots(id) ON DELETE CASCADE,
  voter_id   UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  score      SMALLINT NOT NULL CHECK (score BETWEEN 1 AND 5),
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  CONSTRAINT art_scores_plot_voter_unique UNIQUE (plot_id, voter_id)
);

-- Index for daily vote count query
CREATE INDEX IF NOT EXISTS art_scores_voter_date_idx
  ON art_scores (voter_id, created_at);

-- Index for plot score lookup
CREATE INDEX IF NOT EXISTS art_scores_plot_idx
  ON art_scores (plot_id);

-- Row-level security
ALTER TABLE art_scores ENABLE ROW LEVEL SECURITY;

-- Policy: any authenticated user can read scores
CREATE POLICY "read_art_scores" ON art_scores
  FOR SELECT USING (true);

-- Policy: authenticated users can insert/update their own votes
CREATE POLICY "upsert_own_art_score" ON art_scores
  FOR ALL USING (auth.uid() = voter_id);

-- Average score view
CREATE OR REPLACE VIEW plot_avg_scores AS
  SELECT
    plot_id,
    ROUND(AVG(score)::NUMERIC, 2) AS avg,
    COUNT(*)::INTEGER             AS count
  FROM art_scores
  GROUP BY plot_id;
`;
```

### submitScore

The upsert uses `onConflict: 'plot_id,voter_id'` to update the score if the voter has already rated this plot:

```typescript
export async function submitScore(
  plotId: string,
  voterId: string,
  score: StarScore
): Promise<SubmitScoreResult> {
  // Check if an existing vote is present (to report isUpdate)
  const { count: existingCount } = await supabase
    .from('art_scores')
    .select('id', { count: 'exact', head: true })
    .eq('plot_id', plotId)
    .eq('voter_id', voterId);

  const isUpdate = (existingCount ?? 0) > 0;

  const { error } = await supabase
    .from('art_scores')
    .upsert(
      { plot_id: plotId, voter_id: voterId, score },
      { onConflict: 'plot_id,voter_id' }
    );

  if (error) {
    console.error('[Scores] submitScore error:', error);
    return { success: false, error: error.message };
  }

  return { success: true, isUpdate };
}
```

### getPlotScore

```typescript
export async function getPlotScore(plotId: string): Promise<PlotScore> {
  const { data, error } = await supabase
    .from('plot_avg_scores')
    .select('avg, count')
    .eq('plot_id', plotId)
    .single();

  if (error || !data) {
    // Plot has no votes yet — return defaults
    return { avg: 0, count: 0 };
  }

  return {
    avg: parseFloat(data.avg) || 0,
    count: parseInt(data.count, 10) || 0,
  };
}
```

### getTodayVoteCount

Compute today's midnight in UTC. Supabase filters on `created_at` using ISO 8601 strings:

```typescript
export async function getTodayVoteCount(userId: string): Promise<number> {
  const now = new Date();
  const todayMidnightUTC = new Date(
    Date.UTC(now.getUTCFullYear(), now.getUTCMonth(), now.getUTCDate())
  ).toISOString();

  const { count, error } = await supabase
    .from('art_scores')
    .select('id', { count: 'exact', head: true })
    .eq('voter_id', userId)
    .gte('created_at', todayMidnightUTC);

  if (error) {
    console.error('[Scores] getTodayVoteCount error:', error);
    return 0; // fail open — don't block voting on count errors
  }

  return count ?? 0;
}
```

### Daily vote limit enforcement
The limit of 10 votes per day is enforced in the **UI layer** (T32), not in this module. `scores.ts` is a pure data access layer. The UI calls `getTodayVoteCount` before showing the vote widget and hides it when count ≥ 10. The database does not enforce the 10-vote limit (it only enforces uniqueness per plot-voter pair). This design lets the limit be adjusted without schema changes.

### Error handling philosophy
All three functions return either a result or a safe default on error — they never throw. This prevents a failed vote from crashing the HUD. Errors are logged to the console for debugging.

## Key Constraints
- `submitScore` is an upsert, not an insert — calling it twice for the same `(plotId, voterId)` pair updates the score, not creates a duplicate.
- `getTodayVoteCount` uses UTC midnight as the day boundary, not local midnight — consistent across all time zones.
- `getPlotScore` reads from the `plot_avg_scores` view; this view must exist in the database (run `MIGRATION_SQL` first).
- The `score` parameter is typed as `1|2|3|4|5` — TypeScript prevents passing out-of-range values at compile time.
- Players must not be able to vote on their own plots — enforce this in the UI (T32), not here.
- `MIGRATION_SQL` is a documentation string and is never passed to `supabase.rpc()` or any execution function.

## Acceptance Criteria
- [ ] `submitScore('plot-1', 'user-A', 4)` inserts a row and returns `{ success: true, isUpdate: false }`.
- [ ] Calling `submitScore('plot-1', 'user-A', 5)` a second time updates the score to 5 and returns `{ success: true, isUpdate: true }`.
- [ ] `getPlotScore('plot-1')` returns `{ avg: 4.5, count: 2 }` after two votes of 4 and 5.
- [ ] `getPlotScore` for a plot with no votes returns `{ avg: 0, count: 0 }` without error.
- [ ] `getTodayVoteCount('user-A')` returns 0 at the start of a new UTC day.
- [ ] `getTodayVoteCount('user-A')` returns 1 after one `submitScore` call for that user today.
- [ ] A Supabase network error in `submitScore` returns `{ success: false, error: '...' }` without throwing.
- [ ] `MIGRATION_SQL` is a non-empty string containing `CREATE TABLE art_scores` and `CREATE OR REPLACE VIEW plot_avg_scores`.
