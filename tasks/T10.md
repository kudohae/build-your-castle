# T10 — [Phase 1] — Supabase blocks CRUD

## Overview
This task implements the primary persistence layer for block data — every block placed, moved, or deleted by the player must flow through these functions to survive page reloads. The schema mapping, snake_case-to-camelCase conversion, and upsert semantics are defined here so all other systems have a single reliable interface to work with.

## Scope
Implement `src/supabase/blocks.ts` with four async functions: `loadBlocks(plotId)` fetches all blocks for a plot and maps rows to `Block` objects; `saveBlock(block)` upserts a full block record (id, plot_id, type, x, y, is_wet, health); `updateBlockPosition(id, x, y)` issues a targeted update of only the position columns plus `updated_at`; `deleteBlock(id)` removes a block row. Note that `pixel_mask` is intentionally omitted from the schema interaction in this task — T17 adds it as a targeted modification to this file. All functions are async and use the shared Supabase client.

## Output Files
- `src/supabase/blocks.ts` — all block persistence functions with typed returns

## Implementation Details

### Row-to-Block mapping

The Supabase table uses `snake_case`; the `Block` interface uses `camelCase`. All mapping is centralized in one private helper.

```typescript
// src/supabase/blocks.ts
import { supabase } from './client';
import { Block, BlockType } from '../constants/blocks';

/** Map a raw DB row to the Block interface. */
function rowToBlock(row: Record<string, unknown>): Block {
  return {
    id: row.id as string,
    type: row.type as BlockType,
    x: row.x as number,
    y: row.y as number,
    plotId: row.plot_id as string,
    isWet: row.is_wet as boolean,
    health: row.health as number,
    // pixelMask: added in T17
  };
}
```

### loadBlocks

Fetches all blocks belonging to a plot. Returns an empty array if the plot has no blocks — never throws on a zero-row result. Blocks are ordered by their insertion order (ascending `created_at`) so the renderer processes them in a stable sequence.

```typescript
export async function loadBlocks(plotId: string): Promise<Block[]> {
  const { data, error } = await supabase
    .from('blocks')
    .select('id, type, x, y, plot_id, is_wet, health')
    .eq('plot_id', plotId)
    .order('created_at', { ascending: true });

  if (error) throw new Error(`[blocks.loadBlocks] ${error.message}`);
  return (data ?? []).map(rowToBlock);
}
```

### saveBlock

Uses `upsert` so that both new block creation and full-record updates go through the same function. The `onConflict` target is `id` (the primary key). If `updated_at` column exists in the schema, Supabase triggers will handle it; otherwise set it explicitly.

```typescript
export async function saveBlock(block: Block): Promise<void> {
  const { error } = await supabase
    .from('blocks')
    .upsert(
      {
        id: block.id,
        plot_id: block.plotId,
        type: block.type,
        x: Math.round(block.x),   // round to integers to avoid float drift
        y: Math.round(block.y),
        is_wet: block.isWet,
        health: block.health,
        updated_at: new Date().toISOString(),
        // pixel_mask: added by T17
      },
      { onConflict: 'id' },
    );

  if (error) throw new Error(`[blocks.saveBlock] ${error.message}`);
}
```

### updateBlockPosition

A lightweight update for physics-settled position syncs. Only touches `x`, `y`, and `updated_at` to minimize write amplification when blocks settle after a fall.

```typescript
export async function updateBlockPosition(
  id: string,
  x: number,
  y: number,
): Promise<void> {
  const { error } = await supabase
    .from('blocks')
    .update({
      x: Math.round(x),
      y: Math.round(y),
      updated_at: new Date().toISOString(),
    })
    .eq('id', id);

  if (error) throw new Error(`[blocks.updateBlockPosition] ${error.message}`);
}
```

### deleteBlock

```typescript
export async function deleteBlock(id: string): Promise<void> {
  const { error } = await supabase
    .from('blocks')
    .delete()
    .eq('id', id);

  if (error) throw new Error(`[blocks.deleteBlock] ${error.message}`);
}
```

### Database schema reference

The blocks table structure (pre-existing migration, shown for reference):

```sql
CREATE TABLE blocks (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  plot_id     uuid NOT NULL REFERENCES plots(id) ON DELETE CASCADE,
  type        text NOT NULL CHECK (type IN ('dry_sand','wet_sand','wood_plank','stone','shell')),
  x           integer NOT NULL,
  y           integer NOT NULL,
  is_wet      boolean NOT NULL DEFAULT false,
  health      integer NOT NULL DEFAULT 100 CHECK (health >= 0 AND health <= 100),
  created_at  timestamptz NOT NULL DEFAULT now(),
  updated_at  timestamptz NOT NULL DEFAULT now()
  -- pixel_mask  text  ← added in T17
);

CREATE INDEX blocks_plot_id_idx ON blocks(plot_id);
CREATE INDEX blocks_x_idx ON blocks(x); -- used by chunk range queries
```

## Key Constraints
- `x` and `y` values must be rounded to integers before persistence — Matter.js bodies have floating-point positions, and accumulating float drift over many saves would cause blocks to appear at sub-pixel positions after reload.
- `saveBlock` is intended for full-record writes (new placement + periodic full saves); `updateBlockPosition` is for high-frequency position-only updates after physics settling. Do not use `saveBlock` in a tight loop during physics simulation.
- The `type` column has a database-level `CHECK` constraint — inserting an unknown `BlockType` value will produce a `23514` Postgres error. Validate `block.type` against `BLOCK_PHYSICS` keys before calling `saveBlock`.
- `loadBlocks` returns blocks in `created_at` order, not spatial order. The renderer and physics engine impose their own ordering when processing the returned array.
- `deleteBlock` is permanent; there is no soft-delete. Calling code (T13 block placement) must confirm before deleting.
- T17 modifies this file to add `pixel_mask` handling — the `rowToBlock` function and `saveBlock` payload will be extended there. Do not add `pixel_mask` here.

## Acceptance Criteria
- [ ] `saveBlock(block)` followed by `loadBlocks(block.plotId)` returns an array containing the saved block with all fields intact.
- [ ] `updateBlockPosition(id, 150.7, 488.3)` stores `x: 151, y: 488` (rounded integers).
- [ ] `deleteBlock(id)` causes a subsequent `loadBlocks` to not contain that block.
- [ ] `loadBlocks('nonexistent-plot-id')` returns an empty array without throwing.
- [ ] `saveBlock` with the same `id` twice updates the record rather than creating a duplicate row.
- [ ] TypeScript compiles with zero errors; no `any` in function signatures.
